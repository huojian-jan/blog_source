在计算机行业里，大家的沟通节奏都很快，文档、会议、需求、代码评审、测试和上线过程中，几乎每天都会碰到各种缩写词。

刚入行时，看到满屏的 **TDD、BDD、PRD、CI/CD、API、OAuth、LLM**，很容易一脸懵；工作几年后，又会发现很多缩写虽然眼熟，但未必真的理解它们背后的含义和使用场景。

这篇文章整理了程序员、产品经理、设计师、测试工程师、运维工程师最常见的一批缩写术语。  
这里的“前 50”不是严格意义上的官方排名，而是按**互联网/软件行业日常使用频率**和**跨岗位常见程度**整理，适合作为一份入门级速查表。

---

# 为什么计算机行业这么爱用缩写？

原因很简单：

1. **沟通成本低**  
   与其说 “Application Programming Interface”，不如直接说 “API”。

2. **文档更紧凑**  
   技术文档、产品需求文档、设计稿注释、代码注释里，缩写能大幅减少篇幅。

3. **跨团队形成默认共识**  
   研发、测试、产品、设计、运维都在使用同一套术语体系，缩写就像一种“行业黑话”。

但问题也很明显：  
**知道它的字面全称，不等于真的理解它。**  
所以这篇文章不仅会写出全称，还会解释它的实际用途。

# 计算机行业高频 50 个缩写术语

## 一、产品、设计与项目管理

### 1. PRD
- **全称**：Product Requirements Document
- **中文**：产品需求文档
- **解释**：产品经理输出的核心文档，描述需求背景、目标、功能细节、交互说明和验收标准。
- **常见场景**：`这个需求你先看一下 PRD。`

### 2. PM
- **全称**：Product Manager / Project Manager
- **中文**：产品经理 / 项目经理
- **解释**：这是一个有歧义的缩写。互联网公司里多数情况下指 **产品经理**，部分公司也会指 **项目经理**。
- **常见场景**：`这个需求和 PM 对一下。`

### 3. MVP
- **全称**：Minimum Viable Product
- **中文**：最小可行产品
- **解释**：先做一个“能用、能验证、能上线”的最小版本，用来测试市场反馈。
- **常见场景**：`第一期先做 MVP，别把范围拉太大。`

### 4. UI
- **全称**：User Interface
- **中文**：用户界面
- **解释**：用户能直接看到并操作的界面，包括按钮、布局、颜色、图标等。
- **常见场景**：`这个页面 UI 需要统一一下风格。`

### 5. UX
- **全称**：User Experience
- **中文**：用户体验
- **解释**：不仅是“好不好看”，更关注“好不好用”、流程是否顺畅、用户是否容易理解。
- **常见场景**：`这个弹窗虽然 UI 没问题，但 UX 很差。`

### 6. OKR
- **全称**：Objectives and Key Results
- **中文**：目标与关键结果
- **解释**：一种目标管理方法，用来对齐团队目标和衡量执行结果。
- **常见场景**：`这个季度的 OKR 要和业务目标绑定。`

### 7. KPI
- **全称**：Key Performance Indicator
- **中文**：关键绩效指标
- **解释**：用于衡量工作结果的量化指标，比如 DAU、转化率、故障率等。
- **常见场景**：`这个功能上线后要看 KPI 是否提升。`

### 8. PoC
- **全称**：Proof of Concept
- **中文**：概念验证
- **解释**：先证明一项技术或方案“可行”，再决定是否大规模投入。
- **常见场景**：`我们先做个 PoC，验证一下技术路线。`

### 9. RFC
- **全称**：Request for Comments
- **中文**：征求意见稿 / 技术方案讨论文档
- **解释**：在正式实施前，先把方案写出来，让团队评审和反馈。
- **常见场景**：`这个架构调整先发 RFC。`

### 10. FE
- **全称**：Front End
- **中文**：前端
- **解释**：负责浏览器、App 页面、交互层实现的开发方向。
- **常见场景**：`这个页面逻辑先让 FE 看看。`

### 11. BE
- **全称**：Back End
- **中文**：后端
- **解释**：负责服务端逻辑、数据库、接口、权限、调度等系统能力。
- **常见场景**：`这个问题不是前端，是 BE 接口返回错了。`

## 二、测试与质量保障

### 12. QA
- **全称**：Quality Assurance
- **中文**：质量保证
- **解释**：通常指测试岗位，也指一套保证软件质量的方法和流程。
- **常见场景**：`提测前先找 QA 过一遍。`

### 13. UAT
- **全称**：User Acceptance Testing
- **中文**：用户验收测试
- **解释**：在上线前，由业务方、客户或产品侧确认系统是否满足使用要求。
- **常见场景**：`这周要安排 UAT。`

### 14. TC
- **全称**：Test Case
- **中文**：测试用例
- **解释**：对某个功能点如何测试、输入什么、预期结果是什么的明确描述。
- **常见场景**：`这个场景要补几条 TC。`

### 15. UT
- **全称**：Unit Test
- **中文**：单元测试
- **解释**：针对最小代码单元进行测试，通常由开发编写。
- **常见场景**：`这个模块要把 UT 补齐。`

### 16. E2E
- **全称**：End-to-End
- **中文**：端到端测试
- **解释**：从用户入口一路测到系统出口，覆盖完整业务链路。
- **常见场景**：`发布前要跑一遍 E2E。`

### 17. TDD
- **全称**：Test-Driven Development
- **中文**：测试驱动开发
- **解释**：先写测试，再写实现代码，再重构，是一种强调质量和可验证性的开发方式。
- **常见场景**：`这个核心模块适合用 TDD 来做。`

### 18. BDD
- **全称**：Behavior-Driven Development
- **中文**：行为驱动开发
- **解释**：强调从业务行为和用户场景出发描述需求与测试，常见写法是 Given / When / Then。
- **常见场景**：`这个验收场景可以用 BDD 描述。`

### 19. DDD
- **全称**：Domain-Driven Design
- **中文**：领域驱动设计
- **解释**：围绕业务领域建模，把复杂业务系统拆成清晰的领域边界。
- **常见场景**：`订单系统比较复杂，适合按 DDD 方式拆。`

## 三、研发方法与编码原则

### 20. OOP
- **全称**：Object-Oriented Programming
- **中文**：面向对象编程
- **解释**：以对象、类、封装、继承、多态等概念组织程序。
- **常见场景**：`这个设计明显是 OOP 风格。`

### 21. MVC
- **全称**：Model-View-Controller
- **中文**：模型-视图-控制器
- **解释**：一种经典的软件架构模式，用于分离数据、界面和控制逻辑。
- **常见场景**：`这个老项目还是 MVC 架构。`

### 22. SOLID
- **全称**：五大面向对象设计原则的统称
- **中文**：面向对象设计五原则
- **解释**：包括单一职责、开闭原则、里氏替换、接口隔离、依赖倒置，用来提升代码可维护性。
- **常见场景**：`这个类职责太多，不符合 SOLID。`

### 23. DRY
- **全称**：Don't Repeat Yourself
- **中文**：不要重复自己
- **解释**：相同逻辑不要到处复制，应该抽象复用。
- **常见场景**：`这段代码太重复了，不符合 DRY。`

### 24. KISS
- **全称**：Keep It Simple, Stupid
- **中文**：保持简单
- **解释**：方案越简单越稳定，能简单解决就不要过度设计。
- **常见场景**：`先按 KISS 原则做，不要上来就搞复杂架构。`

### 25. PR
- **全称**：Pull Request
- **中文**：拉取请求 / 合并请求
- **解释**：在 Git 协作中提交代码改动，请求团队审核并合并到目标分支。
- **常见场景**：`代码写完了，已经提 PR 了。`

### 26. CI
- **全称**：Continuous Integration
- **中文**：持续集成
- **解释**：频繁合并代码，并自动执行构建、测试和检查，尽早发现问题。
- **常见场景**：`这个项目的 CI 红了，先别合。`

### 27. CD
- **全称**：Continuous Delivery / Continuous Deployment
- **中文**：持续交付 / 持续部署
- **解释**：把通过验证的版本持续送到可发布状态，甚至自动上线。
- **常见场景**：`我们现在已经做到 CI/CD 了。`

### 28. DevOps
- **含义**：Development + Operations
- **中文**：开发运维协同
- **解释**：强调研发和运维协作，追求更快、更稳定的软件交付。
- **常见场景**：`这个平台是 DevOps 体系的重要一环。`

### 29. SRE
- **全称**：Site Reliability Engineering
- **中文**：站点可靠性工程
- **解释**：通过工程化手段保障系统稳定性、可用性和可恢复性。
- **常见场景**：`这块告警和容量管理通常由 SRE 关注。`

### 30. IaC
- **全称**：Infrastructure as Code
- **中文**：基础设施即代码
- **解释**：用代码管理服务器、网络、权限和云资源，避免手工配置。
- **常见场景**：`新环境直接用 IaC 拉起来。`

## 四、数据、接口与后端开发

### 31. API
- **全称**：Application Programming Interface
- **中文**：应用程序编程接口
- **解释**：不同系统、模块之间通信和调用的标准方式。
- **常见场景**：`这个功能需要调用支付 API。`

### 32. SDK
- **全称**：Software Development Kit
- **中文**：软件开发工具包
- **解释**：一整套开发工具、库、文档和示例代码，方便接入某个平台或服务。
- **常见场景**：`先把登录 SDK 接进去。`

### 33. SQL
- **全称**：Structured Query Language
- **中文**：结构化查询语言
- **解释**：关系型数据库最常见的查询和操作语言。
- **常见场景**：`这个问题我先查一下 SQL。`

### 34. NoSQL
- **中文**：非关系型数据库
- **解释**：泛指不使用传统关系模型的数据库，如 Redis、MongoDB、Cassandra 等。
- **常见场景**：`高并发缓存场景更适合 NoSQL。`

### 35. ORM
- **全称**：Object-Relational Mapping
- **中文**：对象关系映射
- **解释**：把数据库表映射为程序里的对象，减少手写 SQL 的成本。
- **常见场景**：`这个项目大量依赖 ORM。`

### 36. CRUD
- **全称**：Create, Read, Update, Delete
- **中文**：增删改查
- **解释**：最基本的数据操作能力。
- **常见场景**：`这个后台本质上就是一套 CRUD 系统。`

### 37. ETL
- **全称**：Extract, Transform, Load
- **中文**：抽取、转换、加载
- **解释**：数据仓库和数据平台里最经典的数据处理流程。
- **常见场景**：`日报数据是通过 ETL 跑出来的。`

### 38. BI
- **全称**：Business Intelligence
- **中文**：商业智能
- **解释**：通过报表、看板、分析模型帮助业务理解数据并做决策。
- **常见场景**：`这个指标看 BI 看板就行。`

### 39. REST
- **全称**：Representational State Transfer
- **中文**：表述性状态转移
- **解释**：一种常见的 Web API 设计风格，强调资源化和标准化请求方式。
- **常见场景**：`这个接口是典型的 REST 风格。`

### 40. RPC
- **全称**：Remote Procedure Call
- **中文**：远程过程调用
- **解释**：像调用本地函数一样调用远程服务，常见于微服务之间通信。
- **常见场景**：`服务之间走 RPC，不直接暴露 HTTP。`

## 五、网络、安全与身份认证

### 41. HTTP
- **全称**：HyperText Transfer Protocol
- **中文**：超文本传输协议
- **解释**：Web 世界最基础的应用层协议。
- **常见场景**：`这个接口返回的是 HTTP 500。`

### 42. HTTPS
- **全称**：HyperText Transfer Protocol Secure
- **中文**：安全超文本传输协议
- **解释**：在 HTTP 基础上增加加密传输，提升安全性。
- **常见场景**：`正式环境必须全站 HTTPS。`

### 43. DNS
- **全称**：Domain Name System
- **中文**：域名系统
- **解释**：把域名解析成 IP 地址，相当于互联网的“电话簿”。
- **常见场景**：`这个问题可能是 DNS 解析异常。`

### 44. CDN
- **全称**：Content Delivery Network
- **中文**：内容分发网络
- **解释**：把静态资源分发到更靠近用户的节点，加快访问速度。
- **常见场景**：`图片和脚本资源最好走 CDN。`

### 45. SSO
- **全称**：Single Sign-On
- **中文**：单点登录
- **解释**：一次登录后访问多个系统，无需重复认证。
- **常见场景**：`公司内部系统一般都会接 SSO。`

### 46. OAuth
- **全称**：Open Authorization
- **中文**：开放授权
- **解释**：一种授权框架，允许用户授权第三方访问自己的资源，而无需直接暴露密码。
- **常见场景**：`微信登录、GitHub 登录背后常用 OAuth。`

### 47. JWT
- **全称**：JSON Web Token
- **中文**：JSON Web 令牌
- **解释**：一种常见的身份认证令牌格式，经常用于登录态和接口鉴权。
- **常见场景**：`登录成功后服务端会返回 JWT。`

## 六、AI 与新一代计算热点

### 48. AI
- **全称**：Artificial Intelligence
- **中文**：人工智能
- **解释**：让机器具备“感知、理解、决策、生成”等能力的总称。
- **常见场景**：`现在很多产品都在做 AI 能力接入。`

### 49. ML
- **全称**：Machine Learning
- **中文**：机器学习
- **解释**：AI 的重要分支，让系统通过数据训练出模型，而不是手写所有规则。
- **常见场景**：`推荐系统和风控系统里常见 ML。`

### 50. LLM
- **全称**：Large Language Model
- **中文**：大语言模型
- **解释**：近年来最热门的 AI 术语之一，擅长理解和生成自然语言，比如对话、总结、翻译、代码生成等。
- **常见场景**：`这个功能底层接的是 LLM。`

# 这 50 个缩写里，哪些最值得优先记？

如果你刚入行，我建议先优先掌握这 15 个：

- **API**
- **SDK**
- **UI**
- **UX**
- **PRD**
- **PM**
- **QA**
- **TC**
- **TDD**
- **PR**
- **CI/CD**
- **SQL**
- **HTTP / HTTPS**
- **SSO**
- **LLM**

因为它们横跨产品、设计、研发、测试、上线和 AI，是最容易在日常工作中反复碰到的。

# 如何高效记住这些术语？

我的建议是三步：

## 1. 不要死记硬背全称
先记住“它是干什么的”，比记住英文展开更重要。

例如：
- **PRD**：需求文档
- **TC**：测试用例
- **CI**：自动集成构建
- **JWT**：登录认证令牌

理解用途之后，再去记全称会轻松很多。

## 2. 按场景分组记忆
比起按字母顺序记，按工作场景记更有效：

- 做需求会看到：**PRD、PM、MVP、UI、UX**
- 写代码会看到：**API、SDK、OOP、SOLID、PR**
- 测试发布会看到：**QA、TC、UAT、CI、CD**
- 后端和数据会看到：**SQL、ORM、CRUD、ETL**
- 安全与登录会看到：**SSO、OAuth、JWT**
- AI 项目会看到：**AI、ML、LLM**

## 3. 在真实项目里反复使用
缩写不是拿来背单词的，而是拿来工作的。  
当你在 PRD、PR、测试单、接口文档、监控告警、AI 方案里反复看到它们时，自然就记住了。

# 结语

计算机行业的缩写，本质上是一种高密度沟通语言。  
它能提升效率，也会制造门槛。

刚接触时看起来像“天书”，但一旦把它们放回真实工作场景中理解，你会发现这些词其实并不难。  
真正重要的不是记住每个缩写的英文全称，而是知道：

- 它出现在哪个环节
- 它解决什么问题
- 它和哪些角色有关
- 它在项目里怎么用
